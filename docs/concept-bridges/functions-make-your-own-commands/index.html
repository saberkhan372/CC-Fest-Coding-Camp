Functions: Make Your Own Commands

A concept bridge for writing your own functions. Instead of repeating the same drawing steps, you name them once and call them anywhere.

Before this bridge, you might think
To draw a house I write 10 lines of code. To draw two houses I copy those 10 lines. To draw 10 houses I have 100 lines.
After this bridge
I write a drawHouse(x, y, size) function once. To draw 10 houses I call it 10 times with different arguments.

Name it once, use it everywhere

A function is a named block of code that you define once and call many times. Parameters let you customize each call — the same function draws one flower here and a different one there. Return values let functions compute and hand back results for you to use. Composing your sketch from small, named functions makes it easier to read, debug, and change.

Teacher move

Prompt Open the Repeat tab and count the drawing lines. Switch to Define and count again. Now increase the shape count to 8 — ask learners which version they would rather edit and why, before explaining what parameters actually do.
Misconception Students write functions with no parameters and wonder why every copy looks identical. Parameters are slots the caller fills — the function defines what to draw, but each call decides where, how big, or what color. Without parameters, all calls share one fixed answer.
Ask If you change the shape's color inside the function definition, how many call sites are affected? Now imagine you had copied the code instead — how many lines would you need to update? What does that difference tell you about when to write a function?
